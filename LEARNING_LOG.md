### 今日发现的卡点 / 错误(全部已修复)

#### 卡点 1:AI 给的依赖版本号过时(上午发现)
- 现象:AI 给的 fastapi==0.115.0 等版本是 2025 年的旧版
- 实际:2026 当前稳定版 fastapi==0.136.1, uvicorn==0.46.0, jinja2==3.1.6, python-multipart==0.0.27
- 修复:让 AI 联网核查后重写 requirements.txt
- **教训**:AI 给的"具体值"(版本号、链接、API 用法)默认要验证

#### 卡点 2:PyCharm 自动给 .gitignore 加 .py 后缀
- 现象:"New → File" 在 Python 项目里建 `.gitignore`,被加成 `.gitignore.py`,被 PyCharm 当 Python 文件解析,报 39 个语法错误
- 修复:Refactor → Rename 去掉 .py
- **教训**:任何陌生命名的文件,建完都要回头看一眼标签页和图标

#### 卡点 3:PyCharm 终端的"装包提示"误以为要立即操作
- 现象:requirements.txt 被 PyCharm 检测到,弹黄色横幅 "Install requirements"
- 修复:点 "Ignore requirements" 关掉,装包是下一步的事
- **教训**:工具的智能提示有时打乱节奏,要分清"现在做"和"以后做"

#### 卡点 4:.idea/ 被误 commit 进 Git 仓库(晚上踩的大坑)
- 现象:第一次 `git add .` + `git commit` 时,本应被 .gitignore 忽略的 `.idea/` 文件夹被纳入了 staging 和 commit
- 自我警觉:`git status` 看到 staging 里有 `.idea/` 时,主动停下来截图问 AI(没有盲目继续)
- 诊断过程:
  - 用 `git ls-files` 确认追踪列表
  - 用 `git check-ignore -v .idea/misc.xml` 让 Git 自己回答"为什么没忽略"
- 根本原因:**.gitignore 的"先后顺序陷阱"**——已经被追踪的文件,.gitignore 对其无效
- 修复:`git rm -r --cached .idea/`(只从 Git 移除,不删硬盘文件)→ 再 commit 一次"修复:从仓库移除被误纳入的 .idea/ 文件夹"
- **教训**:.gitignore **必须在第一次 git add 之前**就建好;一旦文件已被追踪,需要 `git rm --cached` 显式清理

#### 卡点 5:.gitignore "神秘失效"(晚上踩的怪坑)
- 现象:修复 .idea/ 污染后,再次创建 .idea/ 文件,git status 仍然能看到——.gitignore 完全没起作用
- 自我警觉:主动问 AI"有什么命令能验证 .gitignore 真的生效?"——这个问题救了今晚
- 诊断过程:
  - `git ls-files` 显示历史已清理干净
  - `git check-ignore -v .idea/misc.xml` **无任何输出**(规则没生效)
  - 排查文件编码:PyCharm 显示 UTF-8(编码不是元凶)
  - 怀疑文件中有"看不见的字符"(零宽空格、BOM 等),但没追到具体原因
- 修复:在 PyCharm 里全选删除 .gitignore 内容,粘贴**纯英文版**(无中文注释),保存
- 验证:`git check-ignore -v` 这次正常输出 `.gitignore:9:.idea/  .idea/misc.xml` ✅
- **教训**:配置文件按工程惯例**用英文注释**——技术上中文支持,但避免编码相关的隐形踩坑;**修不到根因时,实用主义重写比追根究底更高效**

#### 卡点 6:commit 中途出错的隐忧(主动求证)
- 现象:某次 commit 时 .gitignore 还在 PyCharm 编辑器里打开,commit 报错;关闭文件后再 commit 成功
- 主动求证:事后问 AI"中途报错对结果有没有影响?"
- 结论:Git commit 是原子操作(要么全成要么全败),没有"半成功"状态;关闭文件后再 commit 完全等同于全新操作
- **教训**:Git 操作是原子的,但**编辑配置文件后必须 Ctrl+S 保存,再去命令行操作**——"我看到的"≠"硬盘上的"
