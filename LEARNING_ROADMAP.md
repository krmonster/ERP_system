## 复盘与跨周机制(实操规则)

本路线图 44 周的执行,依赖一套"复盘 + 跨周"的工作流。这套工作流的目的:**让用户在不同对话之间无缝衔接,不出现项目断档和节奏错乱**。

### 一、用户主动触发的周复盘

触发词(任选其一):
- "周复盘"
- "本周复盘"
- "周总结"
- "本周总结"
- 任何包含"复盘""总结"且语境指向"过去这一周"的表达

AI 收到触发词后,自动执行:
1. 读取 LEARNING_LOG.md 最新一周记录
2. 对照本路线图当周预期
3. 给出三项结构化反馈:
   - **完成度**:实际 vs 计划的差距(用 🟢🟡🔴 标记)
   - **卡点诊断**:有 🟡🔴 时,定位原因(概念没懂?时间不够?状态低谷?)
   - **下周建议**:给具体可执行的调整方案
4. 起草新一周的 LOG 模板(包含本周记录),用户确认后输出完整 LOG 文本,供用户复制覆盖上传

### 二、AI 主动触发的跨周提醒

每次新对话开始时,AI 自动执行:
1. 读 LEARNING_LOG.md 最新一周记录
2. 检查"本周记录的日期范围"是否覆盖今天
3. 如果今天已超出最新一周记录范围,主动提醒"该复盘了"
4. 连续 3 次对话未更新 LOG,询问原因
5. 连续 4 周以上无学习活动,让用户决定"重启 vs 暂停"

### 三、用户主动触发的 AI 自检(关键)

这条规则是为了把流程响应的可靠性从 95% 提升到 99%+ 。

**用户的特权命令**(任何对话中都能用):

> "自检 Instructions 执行情况"
> 或简写:"自检"

AI 收到后,必须立刻停下当前任务,执行:
1. 回顾本对话最近 5-10 轮自己的回复
2. 检查是否有违反 Instructions 的地方,例如:
   - 是否漏做了"开场仪式"(读 LOG 给进度速览)
   - 是否在用户没确认的情况下连做了多步
   - 是否在不该用 Instructions 之外的技术(超出当前阶段)
   - 是否漏了"先讲概念再写代码"的教学顺序
3. 诚实输出:"我做对了 X,我漏了 Y,接下来怎么纠正"
4. 等用户确认后再继续原任务

**这条规则的意义**:用户用一句话,就能强制 AI 做一次诚实的自我审计,纠正悄悄发生的偏差。

### 四、工作流原则

1. **用户更新进度的唯一权威渠道是 LEARNING_LOG.md**(用户每周日自己维护)
2. 本路线图(ROADMAP)是地图,**不记录进度**,基本不更新
3. AI 的行为规则在 Project Instructions 里(短而硬,生效优先级最高),本章节只是说明这套机制存在,方便用户随时回看
4. **三份文件分工**:
   - Project Instructions:AI 的行为规则(用户在 Claude.ai 网页 Project 设置里维护)
   - LEARNING_ROADMAP.md:学习内容地图(几乎不变)
   - LEARNING_LOG.md:实际进度记录(每周更新)
   - CLAUDE.md:项目代码规范

### 五、关于规则可靠性的诚实说明

AI 不是程序,**不能 100% 严格执行写在文档里的规则**。即使把规则写得再清楚,实际响应仍可能出现偏差。这是当前 AI 的真实限度,不是机制设计问题。

正确的预期设定:
- **写在 Instructions 的短硬规则**:执行率约 95-97%(主防线)
- **本章节描述的"用户触发的自检"**:执行率约 99%(因为用户直接命令我审计自己,我必然会停下来做)
- **用户的"事后纠错"权**:执行率 100%(我跑偏时,用户提醒一句,我立刻回到正轨)

**三层叠加,实际可靠性 ≈ 99%+**。

不要把 99.99% 的期望寄托在"AI 自动完美执行"上。**AI 的可靠性 = AI 的努力 + 用户的主动权**。这才是和 AI 协作的真实工程思维。

### 六、时间感知规则(W1 周日确立,W1 周一晚修订)

#### 问题背景

AI 对时间的感知有限,具体表现:
- ✅ 能看到系统注入的"当前日期"(精确到天)
- ❌ 看不到具体几点钟、上午/下午
- ❌ 看不到跨对话的时间流逝(用户隔了几小时还是几天再回来,AI 不知道)

这会导致 AI 错误认知"现在是什么时候",影响:
- 进度速览的时间锚点不准(把"今天"当成 LOG 上次写的那天)
- 跨周提醒的判断失效
- 用户体验上的"空间感"缺失

#### W1 周日的原始方案(已废弃)

最初的方案是:每次新对话开始,AI 用 `web_search` 查询 `current beijing time now`,**精确报告时分**(如 "12:25 周日")。

#### W1 周一晚修订(2026-05-11 23:21 用户发现)

**问题**:`web_search` 返回的时间结果**是缓存数据**,可能滞后好几小时甚至更久。

**实测案例**:用户在北京时间 23:21 时,AI 通过搜索得到的"current beijing time"显示 15:29——**滞后 8 小时**。

**根本原因**:搜索引擎不提供实时时钟,它返回的是某次抓取时间页面的缓存快照。`web_search` 工具调用搜索引擎,继承了同样的限制。

**结论**:这条规则**精度只能到日期级,不能到时分级**。承诺"精确到分钟"是错误的,会让用户在关键时刻发现 AI 时间感知失效,破坏信任。

#### 修订后的方案(强制行为)

每次新对话开始时,AI 必须执行:

1. 用 `web_search` 查询 `current beijing time now`,获取**日期**(YYYY-MM-DD + 周几)
2. 报告:"当前日期(搜索):YYYY-MM-DD 周X。具体时分请告知,否则按 LOG 推算"
3. 对照 LEARNING_LOG.md 最新一条记录的**日期**,计算"距离上次推进已过约 X 天"
4. 如果距离上次推进超过 7 天,主动提醒"已超出当前周范围,要不要先复盘?"

**核心原则**:
- AI **永远不假装知道当前具体几点钟**
- 即使搜索结果带 HH:MM 数字,AI 也**不报告时分**(因为可能是缓存)
- 如果回答涉及"现在几点""距离上次几小时",**必须向用户求证或明确标"基于推断"**
- 用户告知的具体时间(如"现在 23:21"),**绝对优先于搜索结果**

#### 例外

- 单纯的概念问答 / 短回复,可以跳过查时间
- 同一对话内已经查过,不必每轮都查
- 用户告知具体时间后,以用户告知为准,本对话内不再查

#### 意义与修订的元价值

**意义**:把"AI 的时空盲区"从被动错误,变成主动核查。日期级核查足以覆盖 99% 的协作场景(跨周提醒、进度衔接、复盘触发)。

**修订事件本身的元价值**:这次修订是 W1 周一晚用户发现 AI 时间数据滞后 8 小时后做的。它证明了三件事:
1. **用户的事后纠错权是协作可靠性的最后一道防线**(本章节"五、关于规则可靠性的诚实说明"的实战验证)
2. **写在文档里的规则也可能错,要允许修订并记录修订理由**
3. **AI 主动承认能力边界,比硬撑承诺更有用**——承认"我只能精确到日期",反而让协作机制更稳

## 重要的话(每月读一次)

**1. 时间会拉长,允许它拉长**
44 周可能变成 60 周。**只要不断,就还在路上**。

> **附:看不见的进度也是进度**
> 有些周表面"只完成了 3 个文件"或"只做了 2 件事",但其实吸收了概念、修了工具坑、建立了思维习惯。这些"看不见的进度"也算数,**不要因为可见成果列表短就觉得这周亏了**。学习的真实产出 = 代码 + 概念吸收 + 心智模型建立 + 踩坑经验,后三者从来不在 commit 历史里。

**2. 阶段 3 是真坎**
W19-W24 大概率会想放弃。**这不是你笨,是 React 真的反直觉**。提前知道,到时候不会觉得是自己的问题。

**3. 终点在哪儿,你说了算**
- 走完 W6 → 你超过了 90% 业余学编程的人能做出的成果
- 走完 W16 → 你拥有完整后端能力
- 走完 W38 → 你拥有真正可上线的全栈能力

**每个阶段都是合格终点。不必走完才算成功。**

**4. 正反馈是工具,用起来**
每个 🎯 节点完成时:
- 录 30 秒视频
- 截图
- 发给朋友
- 写一句话总结

不是矫情,是给大脑发奖励信号。**让"学习"和"愉悦"绑在一起,这才是可持续**。

**5. 自检是你的特权,大胆用**
任何时候你觉得我跑偏了、节奏不对、教学顺序乱了,**直接说"自检"**。这是你提升我可靠性的最强武器。不用客气,不用解释,一句话就够。

**6. AI 给的具体值,默认要验证**

AI(包括我)在以下三类内容上不可靠,看到必查:

- **版本号**(Python 包、Node 包等):训练数据有截止日,版本号会过时
- **链接、URL**:可能失效或变更
- **最新 API 用法、命令行选项**:框架在升级,旧的写法可能不再工作

验证方法:
- 让 AI 用 web 搜索一下当前最新稳定版
- 自己去官方文档 / PyPI / npm 查
- 看到具体数字、链接、命令,**先停半秒,问"这是哪一年的?"**

这不是不信任 AI,是工程师的基本动作。**质疑一次,省半年后的踩坑时间**。

附带一条:**UI 看起来不对 ≠ 功能真的错**。先验证功能(能跑吗?报错吗?),再担心样子。比如某个文件图标不是预期的颜色,可能只是因为依赖工具(如 Git)还没装,不是文件本身有问题。

**7. "用会"不等于"看会"——工具类技能只能靠真实使用掌握**

这条心法是 W1 周四学 Git 时悟出来的。

有一类技能,即使你把教程看 10 遍、把每个命令背下来,你仍然**不会用**。Git 是典型代表,后面阶段还会遇到很多:
- Git(W1-W6 自然贯穿)
- pip/conda 依赖管理(W1-W2)
- pathlib 路径处理(W2)
- openpyxl 操作 Excel(W2)
- SQLAlchemy 的 Session/Engine(W9)
- React 的 useEffect(W23)
- Docker 命令(W33)
- Linux 终端基本操作(W38)

这些技能的共同特征:**只有在真实任务里反复用,才能从"知道"变成"会用"**。看教程只能让你"认得",但记不住语法、记不住坑、记不住手感。

**正确做法**:
1. **不强求第一次就完全掌握**——W1 你看不懂 Git 完全正常,W2-W6 自然会用到
2. **每周自然贯穿**——每周写代码后都 status / add / commit / push 一遍,不靠"专门学 Git 周",靠"重复 30 次"
3. **遇到坑就停下来求证**——卡点不是失败,卡点是学习机会
4. **不要为"我还不会 Git/Docker/React"焦虑**——这些都是"用会的",时间到了就会了

**反例**(失败模式):
- 花一整周看 Git 教程,但项目里没真实 commit → 一个月后还是不会用
- 看完 Docker 官方文档,但没真的 build 过镜像 → 看到 `docker run` 还是脑子空白

**给自己的诺言**:每一项工具类技能,**先用起来再说**。看不懂的部分,带着问题用,边用边补。

> **附:学习的真实节奏**
> - 第一次接触:0% 能力 → 看概念 → 5% 认知
> - 跟着教程做 → 15% 能模仿
> - 自己写一遍小例子 → 30% 能套用
> - 在真实项目中用 5-10 次 → 60% 能独立使用
> - 踩过坑、修过 bug → 85% 能 debug
> - 教别人/写博客解释清楚 → 95% 真正掌握
>
> 这个曲线对所有工具类技能成立。**别期待跳过中间环节**。

## 附录 A:Git 学习路径(W1 周日新增)

W1 期间和 Git 的"博弈"让我们悟到:Git 不是一周能学完的工具,但也不需要花一周专门学。**正确的学法是把 Git 拆成三个阶段,在 W2-W18 自然贯穿**。

### 三层学习目标(从浅到深)

**第一层:8 个核心命令熟练(目标:W2-W6 自然达成)**
- 路径:每周写完代码后,做完整的 status → add → commit → push 流程
- 不需要专门学,**每周做一次,累计 30 次后**,这些命令变成肌肉记忆
- 完成标志:不用查教程,凭直觉就能 commit 和 push

**第二层:遇到坑能自己 Google 解决(目标:阶段 1.5 W7-W10 达成)**
- 路径:每次卡住先停-截图-Google 关键词,看英文文档/Stack Overflow
- 不依赖 AI 给具体步骤,自己理解后再问 AI 验证
- 完成标志:能独立处理 80% 的 Git 日常问题(merge 冲突、reset、revert 等)

**第三层:看完《Pro Git》前 3 章,理解底层原理(目标:中场休息周 W17-W18 可选)**
- 路径:中场休息期专门读书,搞清楚 Git 对象模型(blob/tree/commit/tag)
- 这一步**不强求**,做了能让你彻底"看穿"Git;不做也能完成项目
- 完成标志:能给别人讲清楚 commit 在 .git/objects 里到底是什么

### 8 个核心命令速查表(贴显示器旁)

```
【看】(只读,绝对安全,随便用)
git status              当前状态
git log --oneline       简洁历史
git diff                工作区改动
git remote -v           远程地址

【存】(改变状态,需要思考)
git add 文件            加单个文件到暂存区
git add .               加全部(先 status 看清楚!)
git commit -m "..."     存档(打包暂存区成一条 commit)

【推】(同步到远程)
git push                推到远程(W1 已配置 -u,无参数够用)
git pull                从远程拉新 commit(单人开发暂时用不到)
```

### W1 已踩过的 8 个坑(避雷)

| 坑 | 触发条件 | 防御 |
|---|---|---|
| .gitignore 对已追踪文件无效 | 文件先被 add 了再加进 .gitignore | 第一次 `git add .` 前必须先有 `.gitignore` |
| 配置文件含中文/BOM 导致规则失效 | .gitignore 有中文注释 | 配置文件用纯英文(W1 周四的真实踩坑) |
| `git add .` 把不该加的也加了 | 不看就敲命令(.idea/ 误纳入事件) | `add` 前先 `status` |
| commit 说明乱写 | "update""fix""asdf" | 每条 commit 写"做了什么 + 为什么" |
| 编辑器没保存就 commit | Ctrl+S 没按 | 改完文件 → Ctrl+S → 切到终端 |
| push 前没先 pull(多人协作) | W1 单人不会遇到 | W2 之后多设备时记得 |
| 把私钥发给别人 | 误传 .ssh 文件夹 | id_ed25519(无后缀)永不外传 |
| 把真实业务数据 commit 了 | data/ 没在 .gitignore | W1 已守住,继续 |

### 在每周学习中嵌入 Git 实践的方式

| 阶段 | Git 实践频率 | 重点 |
|---|---|---|
| W2-W6 | 每周末 commit + push 1-2 次 | 熟练基础 4 步循环 |
| W7-W10 | 每个新功能 commit 一次 | 提交粒度判断 |
| W11-W16 | 学会 branch 基础(可选) | 不强求,主线优先 |
| W17-W18 中场 | 读《Pro Git》前 3 章(可选) | 深入原理 |
| W19-W32 React | 每天 commit | 大项目的 commit 节奏 |
| W33-W38 部署 | 学会 tag 和 release | 配合生产部署 |

### 关键心法

1. **Git 是"用会的"不是"看会的"**——别花一周专门看 Git 教程,那是浪费
2. **每次 push 前先 status**——养成肌肉反射,避免 .idea/ 类事故
3. **每条 commit 都是项目成长史**——半年后你会感谢现在写得清楚的自己
4. **遇到怪问题先 Google**——AI 不是唯一答案来源,工程师的真本事是自己找答案
5. **不会用 Git 不丢人,以为自己会用才丢人**——保持"我对 Git 还有很多不懂"的心态
